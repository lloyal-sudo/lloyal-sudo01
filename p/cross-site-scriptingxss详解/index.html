Cross-Site Scripting（XSS）详解

2026-03-16

一、什么是 XSS

Cross-Site Scripting（跨站脚本攻击）

  • 攻击者往网页里注入恶意 JavaScript 代码，让浏览器执行
  • 浏览器无法区分是“正常代码”还是“恶意代码”，就会直接运行
  • 本质原因：用户输入未经过滤，直接输出到页面 / 存入数据库

二、XSS 能做什么（危害）

  • 偷取 Cookie、Session，冒充用户登录
  • 劫持账号、发帖、删帖、转账
  • 监听用户操作、跳转到钓鱼网站
  • 获取用户浏览器信息、内网信息

三、XSS 三大类型

1. 反射型 XSS（非持久型）

  • 恶意代码放在 URL 参数里
  • 诱导用户点开链接，代码直接在页面执行
  • 常见场景：搜索框、URL 传参直接显示

例子：

1

http://xxx.com/search?keyword=<script>stealCookie()</script>

2. 存储型 XSS（持久型，最危险）

  • 恶意代码存入数据库
  • 所有访问该页面的用户都会中招
  • 常见场景：评论、留言、昵称、个人简介

3. DOM 型 XSS

  • 漏洞出在前端 JS 直接处理用户输入，修改页面 DOM
  • 数据不经过后端、不进数据库，纯前端漏洞

例子：

1
2

var content = location.hash.substr(1);
document.getElementById('msg').innerHTML = content;

四、XSS 根本原因

  1. 信任用户输入：任何输入都可能是恶意的
  2. 未编码、未过滤直接渲染
  3. 前端使用 innerHTML、document.write 等危险操作

五、XSS 解决方法（最实用、最全）

1. 后端：对输出到 HTML 的内容做转义（编码）

把特殊字符换成实体，浏览器只会显示，不会执行：

  • < → &lt;
  • > → &gt;
  • " → &quot;
  • ' → &#x27;
  • & → &amp;

只要输出到 HTML 前统一转义，绝大多数 XSS 直接堵住。

2. 永远不要信任用户输入

  • 对输入做白名单过滤（只允许合法内容）
  • 禁止/过滤：<script>、javascript:、onclick、onload 等

3. 前端避免危险操作

  • 尽量用 textContent，不用 innerHTML
  • 不用 eval()、setTimeout(字符串) 执行用户输入

4. 使用 Cookie HttpOnly（最重要防护之一）

设置 Cookie 时加上 HttpOnly：

  • JS 无法通过 document.cookie 读取
  • 即使有 XSS，也偷不走 Cookie

5. CSP（内容安全策略）

HTTP 头：

1

Content-Security-Policy: default-src 'self'; script-src 'self'

作用：

  • 只允许信任域名加载 JS
  • 禁止内联脚本、禁止 eval
  • 是目前最强的 XSS 防护手段之一

6. 验证码、Referer 校验、Token

防止攻击者构造恶意链接批量利用。

7. 使用成熟框架

Vue、React、Angular 默认对插值内容转义，天然防大部分 XSS。
只有你刻意用 v-html、dangerouslySetInnerHTML 才会有风险。

六、最简总结

  1. XSS = 注入恶意 JS 让浏览器执行
  2. 分三类：反射、存储、DOM，存储型最危险
  3. 根治方案：
    • 输入白名单过滤
    • 输出HTML 转义
    • Cookie HttpOnly
    • 开启 CSP
    • 前端少用 innerHTML
