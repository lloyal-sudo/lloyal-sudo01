Burt Force(暴力破解漏洞)

2026-03-15

暴力破解（Brute Force） 是攻击者通过自动化工具（如 Burp Suite Intruder） 对目标系统的认证入口（登录、验证码、API 密钥等）进行海量尝试，以猜解有效凭证的攻击方式；Burp Suite 是 Web 安全领域最常用的暴力破解工具，核心依赖Intruder 模块实现自动化字典 / 组合攻击。

一、漏洞简介

1. 核心原理

攻击者利用工具批量替换请求中的用户名、密码、验证码、Token等敏感参数，通过分析响应差异（状态码、响应长度、关键词、跳转行为）判断是否猜解成功。

2. 常见攻击场景

登录表单（用户名 + 密码）暴力破解
短信 / 邮箱验证码爆破（4-6 位数字，无次数限制）
API 接口密钥、会话 Token 猜解
后台管理路径、文件目录枚举

3. Burp Suite 暴力破解核心模块（Intruder）

Sniper（狙击手）：单字典，逐个替换单个标记位置（如已知用户名，爆破密码）
Cluster bomb（集束炸弹）：多字典，全组合替换（同时爆破用户名 + 密码，请求量 = 用户数 × 密码数）
Pitchfork（草叉）：多字典，按索引一一对应替换（如用户 1 对应密码 1）
Battering ram（攻城锤）：单字典，同时替换所有标记位置

二、Burp Suite 暴力破解实战步骤（以登录表单为例）

1. 环境准备

启动 Burp Suite，Proxy 模块默认监听127.0.0.1:8080
浏览器配置代理（与 Burp 一致），关闭浏览器代理自动检测

2. 捕获登录请求

浏览器访问目标登录页，输入任意用户名 / 密码（如admin/123456）
Burp Proxy→Intercept 开启拦截，提交登录，捕获 POST 请求
右键请求→Send to Intruder（快捷键 Ctrl+I）

3. 配置攻击位置（Positions）

清除默认标记（点击Clear §）
选中username和password参数值，点击Add §标记为爆破位置
攻击类型选择：已知用户名选Sniper；未知用户名选Cluster bomb

4. 加载攻击字典（Payloads）

Payload Set 1（用户名）：选择Simple list，加载用户名字典（如users.txt）
Payload Set 2（密码）：选择Simple list，加载密码字典（如rockyou.txt、Top1000 弱口令）
可选：设置Payload Processing（如大小写转换、添加后缀）

5. 攻击优化（Options）

Throttle：设置请求间隔（如 1000ms），避免触发 WAF/IP 封禁
Grep - Match：添加失败关键词（如 “登录失败”“错误”），快速筛选成功请求

6. 启动攻击与结果分析

点击Start attack，Intruder 自动发送所有组合请求
结果判断：
状态码异常（如302跳转、200成功）
响应长度与其他请求差异显著
无失败关键词匹配
验证成功凭证，完成登录

三、暴力破解漏洞解决方法（防御 + 修复）

1. 服务端核心防护（最有效）

（1）登录失败限制（Rate Limiting）
单账户：连续失败 5-10 次，锁定账户 15-30 分钟
单 IP：单位时间（如 1 分钟）请求超阈值，临时封禁 IP
结合Cookie+UA+IP多维识别，防止 IP 伪造绕过
（2）强制验证码机制
登录失败≥3 次，触发图形 / 滑动 / 行为验证码（如 Google reCAPTCHA）
验证码有效期短（≤5 分钟）、一次性使用、后端校验（禁止前端校验）
短信 / 邮箱验证码：6 位数字 + 有效期≤5 分钟 + 次数限制（每日≤5 次）
（3）强化密码策略
强制密码复杂度：≥12 位 + 大小写 + 数字 + 特殊字符
禁止弱口令（如123456、admin、生日、姓名拼音）
密码定期更换（90 天），禁止重复使用历史密码
（4）Token 与会话安全
登录请求使用一次性 CSRF Token（后端生成、前端隐藏、每次请求刷新）
会话 Token高熵值、随机生成、有效期短，禁止在 URL 中传递
（5）WAF 与安全设备
部署 Web 应用防火墙（WAF），识别并拦截暴力破解流量
启用异常请求检测（高频、字典特征、UA 异常）

2. 前端与用户层面防护

前端禁用自动填充、禁止记住密码（敏感系统）
提示用户不重复使用密码、启用双因素认证（2FA）
登录页添加防机器人检测（如滑块、点击验证）

3. 漏洞修复步骤（操作流程）

漏洞定位：通过渗透测试 / 日志分析，确认存在暴力破解的入口（登录、验证码、API）
策略配置：
后端添加登录失败计数 + 账户锁定逻辑
集成验证码服务（如阿里云、腾讯云验证码）
配置WAF 规则拦截高频请求
代码加固：
后端校验所有参数，禁止前端信任
验证码 / Token后端生成 + 销毁，不暴露于前端源码
测试验证：
使用 Burp Suite 重新尝试爆破，验证防护生效
模拟高频请求，确认 IP / 账户锁定机制正常
日志监控：开启登录日志，实时监控异常登录行为，及时告警

四、关键注意事项

暴力破解属于非法攻击行为，仅可在授权测试环境中使用
防御需多维度组合（失败限制 + 验证码 + 2FA），单一措施易被绕过
字典质量决定攻击效率，优先使用弱口令字典 + 社工字典组合

验证码绕过（on server）完整步骤流程

这里指服务端校验不严导致的验证码绕过，比如验证码未过期、可复用、未绑定会话等场景，以 Pikachu 靶场为例。

一、漏洞原理

服务端在验证验证码时存在缺陷：
验证码未销毁/未过期，可重复使用
验证码未与当前会话绑定，攻击者可复用他人有效验证码
验证码逻辑校验不严（如固定值、弱复杂度）
验证码未在每次请求后刷新

攻击者可以：

获取一次有效验证码
暴力破解用户名/密码，复用同一个验证码
绕过验证码防护，完成暴力破解

二、工具准备

Burp Suite（中文版）
浏览器（Chrome/Firefox）
Pikachu 靶场：http://127.0.0.1/pikachu/vul/burteforce/bf_server.php

三、完整攻击步骤

步骤 1：获取有效验证码

  1. 打开 Pikachu「验证码绕过（on server）」页面

  2. 输入任意用户名、密码，手动输入一次正确的验证码并提交

  3. Burp 拦截到该登录请求包，包含参数：

    username=xxx&password=xxx&vcode=xxxx&submit=Login

  4. 记录下当前有效的 vcode 值（比如 8848），这个验证码在服务端未销毁，可重复使用。

步骤 2：将请求发送到 Intruder（入侵者）

  1. 在 Burp 中右键该请求 → 发送到入侵者
  2. 进入 位置 标签页：
  • 点击「清除§」
  • 选中 username 和 password 的值，分别点击「添加§」标记为爆破点
  • 验证码 vcode 保持刚才获取的有效值，不做标记
  • 攻击类型选择：集束炸弹（Cluster bomb）

步骤 3：配置载荷（字典）

  1. 进入 载荷 标签页：
  • 载荷集 1（用户名）：
    • 类型：简单列表
    • 添加常见用户名：admin、test、user、pikachu
  • 载荷集 2（密码）：
    • 类型：简单列表
    • 添加弱口令：123456、admin、123、123123、password

步骤 4：设置成功/失败判断规则

  1. 进入 选项 标签页
  2. 找到 匹配提取（Grep - Match）：

  • 取消所有默认勾选

  • 点击「添加」，输入 Pikachu 登录失败提示：

    username or password is not exists

  • 点击「确定」

  1. （可选）调整 请求引擎：
  • 并发请求数设为 5~10，避免过快被拦截

步骤 5：启动攻击 & 分析结果

  1. 等待攻击完成后，查看结果表：

匹配提取 列：
✅ 打勾 = 包含失败提示 = 登录失败
❌ 未打勾 = 登录成功！
同时看 长度 列，成功请求的响应长度通常与失败请求不同

  1. 找到未打勾的行，对应的 username 和 password 即为正确凭证。

步骤 6：验证漏洞

  1. 用爆破出的账号密码，复用同一个验证码手动登录
  2. 若成功登录，证明服务端验证码未销毁/可复用，漏洞存在。

四、漏洞修复方法

  1. 验证码一次性使用：验证后立即销毁，不允许重复提交
  2. 设置有效期：验证码有效期设为 1~3 分钟，超时自动失效
  3. 每次请求刷新验证码：登录失败/页面刷新后，强制生成新验证码
  4. 增加复杂度：使用字母+数字+干扰线的复杂验证码，避免 OCR 识别

五、关键注意事项

核心是复用有效验证码，所以 vcode 参数不要标记为爆破点
若服务端做了 IP 限流，可适当降低并发请求数
该攻击仅在服务端验证码校验不严的场景下生效，若验证码已销毁则无法绕过

验证码绕过（on client）完整步骤流程

这里指前端校验不严导致的验证码绕过，即验证码仅在浏览器端做校验，服务端未做验证，攻击者可直接绕过前端逻辑发起请求。

一、漏洞原理

验证码仅在前端 JavaScript 中完成校验：

  • 验证码正确/错误的判断逻辑写在前端 JS 里，服务端未校验
  • 攻击者可通过禁用 JS、修改前端代码、直接发包等方式，绕过验证码校验
  • 本质是前端安全不可信，所有校验必须在服务端完成

三、完整攻击步骤

步骤 1：观察前端验证码逻辑

  1. 打开 Pikachu「验证码绕过（on client）」页面
  2. 输入任意用户名、密码，故意输错验证码，点击登录
  3. 观察到页面弹出提示（如“验证码错误”），但页面未刷新、未向后端发请求
  4. 按 F12 打开开发者工具 → 「源代码/ Sources」，找到验证码校验 JS 代码，确认校验逻辑在前端

步骤 2：绕过前端校验（两种常用方法）

方法一：禁用 JavaScript（最简单）

  1. 按 F12 → 点击右上角「设置（⚙️）」
  2. 找到「Debugger」→ 勾选 Disable JavaScript（禁用 JavaScript）
  3. 刷新页面，此时前端验证码校验逻辑失效
  4. 输入任意用户名、密码，验证码随便填（甚至留空），点击登录

方法二：直接抓包修改（推荐用于爆破）

  1. 浏览器代理配置为 Burp，开启拦截
  2. 输入任意用户名、密码、验证码（随便填），点击登录
  3. 若前端拦截导致无法提交，先按方法一禁用 JS，再提交
  4. Burp 拦截到登录请求包，参数格式：
    username=xxx&password=xxx&vcode=xxxx&submit=Login
  5. 右键 → 发送到入侵者（Intruder）

步骤 3：配置 Intruder 爆破

  1. 进入 位置 标签页：
  • 点击「清除§」
  • 选中 username 和 password 的值，分别点击「添加§」标记为爆破点
  • vcode 保持任意值（如 1234），不做标记
  • 攻击类型选择：集束炸弹（Cluster bomb）
  1. 进入 载荷 标签页：
  • 载荷集 1（用户名）：
    • 类型：简单列表
    • 添加常见用户名：admin、test、user、pikachu
  • 载荷集 2（密码）：
    • 类型：简单列表
    • 添加弱口令：123456、admin、123、123123、password
  1. 进入 选项 标签页：
  • 找到 匹配提取（Grep - Match）
  • 取消所有默认勾选，点击「添加」，输入登录失败提示：
    username or password is not exists
  • （可选）调整 请求引擎：并发请求数设为 5~10

步骤 4：启动攻击 & 分析结果

  1. 点击右上角 开始攻击
  2. 等待攻击完成后，查看结果表：
  • 匹配提取 列：
    • ✅ 打勾 = 包含失败提示 = 登录失败
    • ❌ 未打勾 = 登录成功！
  • 同时看 长度 列，成功请求的响应长度通常与失败请求不同
  1. 找到未打勾的行，对应的 username 和 password 即为正确凭证

步骤 5：验证漏洞

  1. 用爆破出的账号密码，随便填一个验证码（甚至错误验证码）
  2. 若成功登录，证明验证码仅在前端校验，服务端未做验证，漏洞存在

四、漏洞修复方法

  1. 服务端校验验证码：所有验证码校验逻辑必须在服务端实现，禁止依赖前端校验
  2. 验证码一次性使用：验证后立即销毁，不允许重复提交
  3. 绑定会话与验证码：验证码与当前 Session 绑定，防止跨会话复用
  4. 验证码有效期：设置 1~3 分钟有效期，超时自动失效
  5. 移除前端校验提示：前端仅做格式提示，最终判断由服务端返回

五、关键注意事项

  • 核心是绕过前端 JS 校验，禁用 JS 或直接发包是最直接的方式
  • 若服务端未校验验证码，vcode 参数可以是任意值，无需正确
  • 该漏洞本质是前后端职责不清，是最基础的安全设计错误

Token 防爆破场景下的暴力破解流程

这里指服务端生成 CSRF Token 防护表单，但防护不严导致可被绕过或批量爆破的场景，以 Pikachu 靶场为例。

一、漏洞原理

正常情况下，表单会携带一次性 CSRF Token：

  • 服务端生成 Token，嵌入表单隐藏域
  • 每次提交需携带有效 Token，防止批量请求
  • 若防护不严（Token 可复用、未绑定会话、可预测、未刷新），攻击者可通过获取 Token → 复用/批量获取 → 暴力破解绕过防护

二、工具准备(同上)

三、完整攻击步骤

步骤 1：分析 Token 位置与特性

  1. 打开 Pikachu「token防爆破?」页面，按 F12 查看源码
  2. 找到登录表单中的隐藏 Token 参数：

html

  1. 手动提交一次登录，观察 Burp 数据包：

username=xxx&password=xxx&token=xxxxxx&submit=Login

  1. 验证 Token 特性：是否每次刷新页面都会变化、是否可重复使用。

步骤 2：配置 Burp 抓取 Token（关键步骤）

  1. 浏览器代理配置为 Burp，开启拦截
  2. 访问登录页面，Burp 拦截到 GET 请求（获取登录页）
  3. 右键该 GET 请求 → 发送到宏（Macro）
  • 新建宏，命名为 Get Token
  • 选择该 GET 请求，确认后保存
  • 配置宏的响应解析：提取 token 值（使用正则表达式 value="([a-zA-Z0-9]+)" 匹配）
  1. 回到登录请求，将其发送到 入侵者（Intruder）

步骤 3：配置 Intruder 爆破位置

  1. 进入 位置 标签页：
  • 点击「清除§」
  • 选中 username 和 password 的值，分别点击「添加§」标记为爆破点
  • token 参数保持原样，不做标记
  • 攻击类型选择：集束炸弹（Cluster bomb）
  1. 进入 选项 标签页：
  • 找到 会话处理规则（Session Handling Rules）
  • 新建规则：
    • 规则动作：运行宏
    • 选择刚才创建的 Get Token 宏
    • 作用范围：当前 Intruder 攻击
    • 作用时机：每次请求前，自动获取新 Token 并更新到请求中

步骤 4：配置载荷（字典）与成功判断

  1. 进入 载荷 标签页：
  • 载荷集 1（用户名）：
    • 类型：简单列表
    • 添加常见用户名：admin、test、user、pikachu
  • 载荷集 2（密码）：
    • 类型：简单列表
    • 添加弱口令：123456、admin、123、123123、password
  1. 进入 选项 标签页：
  • 找到 匹配提取（Grep - Match）
  • 取消所有默认勾选，点击「添加」，输入登录失败提示：
    username or password is not exists
  • （可选）调整 请求引擎：并发请求数设为 5~10，避免 Token 刷新过快

步骤 5：启动攻击 & 分析结果

  1. 点击右上角 开始攻击
  2. 攻击过程中，Burp 会自动为每个请求获取新的有效 Token
  3. 等待攻击完成后，查看结果表：
  • 匹配提取 列：
    • ✅ 打勾 = 包含失败提示 = 登录失败
    • ❌ 未打勾 = 登录成功！
  • 同时看 长度 列，成功请求的响应长度通常与失败请求不同
  1. 找到未打勾的行，对应的 username 和 password 即为正确凭证

步骤 6：验证漏洞

  1. 用爆破出的账号密码，手动提交登录（需携带新 Token）
  2. 若成功登录，证明Token 防护机制可被自动化绕过，漏洞存在。

四、漏洞修复方法

  1. 严格绑定会话与 Token：Token 与当前 Session 强绑定，不同会话不可复用
  2. Token 一次性使用：验证后立即销毁，每次请求必须生成新 Token
  3. 设置有效期：Token 有效期设为 1~3 分钟，超时自动失效
  4. 高熵值生成：使用加密安全的随机数生成 Token，避免可预测
  5. 服务端严格校验：每次提交必须验证 Token 有效性，无效则拒绝请求

五、关键注意事项

  • 核心是自动化获取/更新 Token，通过 Burp 宏实现“请求前获取新 Token”
  • 若 Token 绑定会话，需确保宏与攻击使用同一会话（Cookie 一致）
  • 若服务端 Token 校验严格（一次性+会话绑定），则无法绕过，暴力破解失效
